## 一、项目概述

**目标：** 基于飞书 wiki 文档，构建可持续自动更新的 DMP 平台帮助中心。

**技术栈：** Docusaurus 3.7 + GitHub Pages + GitHub Actions + 飞书 Open API

**线上地址：** https://greycheer.github.io/dmp-help-doccenter/

## 二、需求规格

### 2.1 内容同步

| 需求 | 说明 |
|------|------|
| 数据源 | 两份飞书文档：管理员手册 + 开发者手册 |
| 同步方式 | `npm run sync` 一键拉取，GitHub Actions 自动部署 |
| 内容保真 | 图文与源文档一致，不修改原文内容 |
| 增量更新 | 检测飞书文档变更，仅同步有变化的部分（TODO: 后续迭代） |

### 2.2 文档拆分（核心改动）

**现状问题：** pipeline.js 只按 H2 拆分，一整个大章节塞进一个文件，层级扁平。

**改造目标：** 按叶子节点递归拆分，支持多级目录嵌套。

**叶子节点定义：**
- 某个标题下**不再包含子标题**（只有正文/列表/图片等），则为叶子节点 → 独立 `.md` 文件
- 某个标题下**包含子标题**，则作为目录层级 → 子目录 + `_category_.json`

### 2.3 目录导航

| 需求 | 说明 |
|------|------|
| 侧边栏 | 按文档树结构自动生成，使用 `autogenerated` 模式 |
| 多级目录 | 通过子目录 + `_category_.json` 实现层级嵌套 |
| 排序控制 | frontmatter 的 `sidebar_position` 控制顺序 |
| 折叠展示 | Docusaurus 默认支持，目录层级多时自动折叠 |

### 2.4 图片物料维护

| 需求 | 说明 |
|------|------|
| 下载 | 同步时自动从飞书下载所有图片 |
| 增量维护 | 对比飞书图片列表与本地已有文件，新增的下载，删掉的清除 |
| 目录隔离 | 图片按手册分目录（`static/img/admin-manual/`、`static/img/developer-manual/`） |

### 2.5 搜索 & 部署

| 需求 | 说明 |
|------|------|
| 全文搜索 | `@easyops-cn/docusaurus-search-local`，客户端本地运行，无需外部服务 |
| 自动部署 | 推送到 GitHub 后，Actions 自动构建部署到 GitHub Pages |
| 响应式 | Docusaurus 内置移动端适配 |

## 四、pipeline.js 改造设计

### 4.1 整体流程（改造后）

```
飞书 API
  │
  ├── 1. 认证（getToken）              ← 不变
  ├── 2. 拉取 blocks（getAllBlocks）    ← 不变
  ├── 3. blocks → Markdown             ← 【增强：格式兼容处理，见 4.5】
  ├── 4. 构建标题树（buildHeadingTree）  ← 【新增，含跳级容忍】
  ├── 5. 递归拆分（splitByLeafNodes）   ← 【新增，替换 splitMarkdownByH2】
  ├── 6. 图片增量维护（syncImages）     ← 【新增】
  └── 7. 写文件（writeTreeToDocs）      ← 【新增，替换 buildDocFiles】
```

### 4.2 标题树数据结构

```javascript
// 标题树节点
{
  level: 2,                    // 标题级别 (H2=2, H3=3, ...)
  title: "Product Development", // 标题文本
  slug: "product-development",  // URL 友好的 slug
  blocks: [],                   // 此标题下的非标题 blocks（正文内容）
  children: [],                 // 子标题节点（递归结构）
}

// 完整标题树 = 根节点，children 是 H2 列表
// H2.children = H3 列表
// H3.children = H4 列表
// ...以此类推
```

### 4.3 核心函数设计

#### buildHeadingTree(blocks, startLevel)
- 输入：飞书 blocks 数组、起始拆分级别（H2）
- 遍历 blocks，遇到标题 block 则创建新节点
- 非标题 block 归入当前节点的 `blocks`
- **跳级容忍**：使用栈模式构建父子关系，当标题 level 大于当前 level + 1 时（如 H3 下直接跟 H5），仍视为当前节点的直接子节点，不创建中间空节点
- **空标题过滤**：`extractRichText(elements)` 返回空字符串的标题 block 直接跳过，不创建节点（飞书文档中常见的空白标题残留）

#### isLeafNode(node)
- `node.children.length === 0` → 是叶子节点
- 叶子 → 生成 `.md` 文件
- 非叶子 → 生成子目录

#### writeTreeToDocs(tree, outputDir, imgBaseRelPath)
- 递归遍历标题树
- 非叶子节点 → 创建子目录 + `_category_.json`
- 叶子节点 → 创建 `.md` 文件（frontmatter + blocksToMarkdown 转换内容）
- 每个文件带序号前缀保证排序

#### syncImages(allImageTokens, imgDir)
- 扫描飞书文档中所有图片 token → 集合 A
- 扫描本地 static/img/{slug}/ 已有文件 → 集合 B
- `A - B` → 新增图片，从飞书下载
- `B - A` → 过期图片，删除本地文件
- 下载时带 200ms 间隔，避免频率限制

### 4.4 清理策略

同步开始时，先清空目标目录（docs/{slug}/）的所有 .md 和子目录，再重新生成。保证本地文件与飞书文档完全一致，不会残留旧文件。

### 4.5 blocksToMarkdown 格式兼容处理

当前 `blocksToMarkdown()` 在转换飞书 blocks 时存在多项格式丢失或失真问题，改造时一并修复。以下为增强项清单：

#### (a) 空标题跳过（P0）

**现状**：飞书文档中存在空内容标题 block（编辑残留），转换后输出 `###  ` 等空标题行，会错误触发拆分逻辑并生成无文本 TOC 条目。

**修复**：标题处理分支中，对 `extractRichText(elements).trim()` 结果为空的直接 `continue` 跳过。

#### (b) 裸 URL → 图片检测（P0）

**现状**：飞书中部分图片以纯文本 URL 粘贴（非飞书图片 block），转换后显示为不可点击的纯文本，图片内容丢失。

**修复**：在 `extractRichText()` 中检测匹配图片 CDN 域名的 URL（`qpic.cn`、`feishu.cn`、`larksuite.com` 等），自动转为 `![image](URL)` 语法。

#### (c) 下划线处理修正（P0）

**现状**：所有下划线文本被转成 `**bold**`，导致导航路径等有下划线的内容被误加粗。

**修复**：忽略下划线样式，不插入任何 Markdown 标记（Markdown 原生不支持 `<u>`，加粗语义不对）。如需保留下划线视觉效果，可用 HTML `<u>` 标签。

#### (d) 列表嵌套 depth 支持（P1）

**现状**：飞书列表 block 包含 `depth` 字段表示缩进层级，但 pipeline 忽略了该字段，所有列表项平铺为一级。

**修复**：读取 `data.depth`（默认 0），按 `'  '.repeat(depth)` 前缀缩进，输出标准 Markdown 嵌套列表。

#### (e) callout → Docusaurus admonition（P1）

**现状**：飞书 callout（高亮块）根据背景色区分类型（info/warning/tip/error），但 pipeline 只保留 emoji 并转为 `>` 引用块，丢失了类型语义。

**修复**：根据 `data.style.background` 颜色映射到 Docusaurus admonition 类型，输出 `:::note` / `:::warning` / `:::tip` / `:::danger` 语法。

#### (f) 代码块内容纯文本化（P1）

**现状**：代码块内容经过 `extractRichText()` 处理，可能包含 `**bold**`、`*italic*` 等 Markdown 格式标记，导致代码块渲染异常。

**修复**：代码块分支直接提取 `text_run.content` 纯文本拼接，不经过 `extractRichText()` 的格式化处理。

#### (g) 移除 `< >` 误转义（P2）

**现状**：后处理正则 `(?<!=)<(?![\/!a-zA-Z])` 试图转义 `<` 防止 HTML 注入，但排除条件不精确，导致 `<**Product Firmware Management>**` 等乱码。

**修复**：直接删除该后处理。Docusaurus 3.7 的 MDX 引擎可正确处理文本中的裸 `<` 和 `>`，无需手动转义。

## 七、风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 飞书图片 token 失效 | 图片无法下载 | 每次同步前重新拉取 blocks，获取最新 token |
| 标题层级不规范（跳级） | 目录层级混乱 | `buildHeadingTree` 使用栈模式容忍跳级，不依赖编号 |
| 空标题 block 残留 | 错误拆分 + 空白 TOC | `buildHeadingTree` 中过滤空标题（见 4.5a） |
| 图片以 URL 文本粘贴 | 图片丢失 | `extractRichText` 自动检测 CDN URL 转图片语法（见 4.5b） |
| 列表嵌套信息丢失 | 层级扁平化 | 读取飞书 `depth` 字段输出嵌套列表（见 4.5d） |
| 超长文件名 | 路径超限 | slugify 限制 80 字符 |
| 飞书 API 频率限制 | 下载失败 | 请求间隔 200ms，失败重试 3 次 |
